# AlfaRank.com Development Brief

## 1. Project Direction

AlfaRank.com is being rebuilt as a full corporate website for a modern digital systems company.

The site must present AlfaRank as a company that designs, builds, integrates, and launches digital systems for business:

- web platforms;
- AI automation;
- SEO and content infrastructure;
- data systems;
- scraping and monitoring tools;
- e-commerce infrastructure;
- video and content generation pipelines;
- CRM and lead automation;
- internal business tools;
- WordPress and API integrations.

The website is not a founder portfolio, not a one-page landing page, and not a standard SEO agency website.

## 2. Positioning Rules

Do not use the concept of evolution from SEO agency to AI/dev company.

Avoid any framing such as:

- old website;
- previous background;
- before / now;
- we grew from SEO;
- new generation of AlfaRank;
- rebranding story;
- personal founder-led positioning.

The site should feel as if AlfaRank has always been a modern digital systems company.

Core positioning:

> AlfaRank designs and launches digital systems for business operations, content infrastructure, automation, and growth.

Alternative short positioning variants:

- AlfaRank designs and launches digital systems for businesses that need automation, scalable content infrastructure, data workflows, and web platforms.
- AlfaRank builds practical digital systems: web platforms, AI automation, SEO/content infrastructure, data tools, and business integrations.
- AlfaRank helps companies turn operational, marketing, and data processes into working digital systems.
- AlfaRank is a digital systems company focused on automation, web infrastructure, content operations, and data-driven business tools.
- AlfaRank builds the infrastructure behind modern business growth: platforms, workflows, integrations, automation, and data systems.

## 3. Site Tone

The tone must be:

- technological;
- confident;
- corporate;
- practical;
- implementation-focused;
- business-output-focused.

Avoid:

- startup noise;
- hype language;
- "young team";
- "individual approach";
- "full range of services";
- generic SEO agency wording;
- personal portfolio tone;
- overpromising claims.

## 4. Information Architecture Principle

The site should not be structured as a flat list of services.

The structure must be based on:

- capabilities;
- solutions;
- systems;
- projects;
- technologies;
- industries.

Main logic:

- Capabilities = what AlfaRank can build.
- Solutions = which business problems AlfaRank solves.
- Systems = what types of systems/products AlfaRank can show.
- Technologies = how the systems are implemented.
- Industries = where these systems are applicable.
- About = how the company thinks and works.
- Start Project = project intake.

The home page should be assembled later as a navigation hub across these sections.

## 8. Page Structure Patterns

### Capability Page Pattern

Each capability page should answer:

- what AlfaRank builds;
- where it fits in business;
- typical system components;
- inputs and outputs;
- relevant technologies;
- related solutions;
- related systems/projects;
- project start CTA.

Example structure:

```text
Hero
What this capability covers
Systems AlfaRank can build in this area
Typical modules
Business outputs
Related solutions
Related systems/projects
Technology layer
Start a project CTA
```

### Solution Page Pattern

Each solution page should translate a business problem into a system.

Example structure:

```text
Business problem
What system solves it
Core modules
Implementation options
Operational output
Relevant capabilities
Related systems/projects
Start a project CTA
```

### System / Project Page Pattern

Systems should not be shown as generic portfolio cards. Use system profiles.

Example structure:

```text
System overview
Business context
Core modules
Workflow
Technology layer
Outputs
Use cases
Related capabilities
Start a project CTA
```

## 9. Home Page Logic

The home page is a navigation hub, not a long promotional landing page.

Recommended flow:

1. Hero: short company positioning and direct CTA.
2. What AlfaRank Builds: cards for main capabilities.
3. Business Problems We Solve: links to solution pages.
4. Systems & Projects: links to system profiles.
5. Technology Layer: stack and implementation areas.
6. Industries: business types where systems apply.
7. How Projects Start: business problem -> scope -> prototype -> implementation -> integration -> measurement.
8. Final CTA: Start a project.

## 10. Technology Stack

Use the same stack pattern as the other company sites:

- Next.js;
- Astro where it fits static/content-heavy sections;
- GitHub for repository and deployment workflow;
- Cloudflare Pages for hosting/deployment.

Initial implementation recommendation:

- Use Astro as the primary static site shell if the first version is mostly corporate/content pages.
- Use React/Next.js patterns or islands only where interactive components are needed.
- Keep the architecture compatible with future migration or partial use of Next.js if product-style pages, dashboards, or advanced interactive modules are added.

Deployment assumptions:

- repository on GitHub;
- production build deployed to Cloudflare Pages;
- preview deployments from branches or pull requests;
- content and route structure should remain Git-friendly and easy to review.

## 11. Design Direction

The design must start from architecture and content, not from visual decoration.

General direction:

- modern corporate digital systems company;
- technological but restrained;
- clear navigation and page hierarchy;
- cards, grids, and sections should support scanning;
- no loud startup aesthetics;
- no generic SEO agency look;
- no personal portfolio look.

## 12. Liquid Glass Visual System

The visual direction can use the liquid glass system being developed in the Vidorix-related folder/project.

Important: the background system must be replaceable later.

Background direction:

- blue color range;
- wave-like liquid shapes;
- multiple blue/cyan/navy tones;
- soft depth, not aggressive decoration;
- should work behind transparent glass cards;
- should be implemented as a reusable background layer/component;
- should be easy to swap with a different generated background later.

Recommended implementation idea:

```text
Base layout
├── replaceable background component
├── page content wrapper
├── transparent glass sections/cards
└── normal readable typography layer
```

Do not hardcode the background into every page. Create a single reusable background component or layout slot.

## 13. Cards and Surfaces

Card direction:

- maximum transparency where readability allows;
- glass-like surfaces;
- straight or nearly straight corners;
- no strongly rounded cards;
- no nested decorative cards;
- thin borders;
- subtle background blur;
- controlled shadows;
- clear typography contrast.

Suggested baseline:

```css
border-radius: 4px;
background: rgba(255, 255, 255, 0.08);
border: 1px solid rgba(255, 255, 255, 0.18);
backdrop-filter: blur(18px);
```

The exact values can be adjusted during implementation, but the design principle should remain:

- transparent;
- sharp;
- technical;
- readable;
- replaceable background.

## 14. Contact / Project Intake

The Start a Project page should collect:

- project type;
- business problem;
- current website/system;
- desired result;
- optional budget;
- optional timeline;
- contact details.

The form should help classify the lead by system type, not just collect a generic message.

## 15. Immediate Build Priorities

First implementation phase:

1. Set up project stack and deployment-ready structure.
2. Create global layout and navigation.
3. Create replaceable liquid glass background component.
4. Create reusable transparent card/section components.
5. Build first-version pages from the sitemap.
6. Add project intake form structure.
7. Prepare content placeholders that follow the positioning rules.

Visual detail, animations, additional system pages, and industry pages can be refined after the architecture is working.
